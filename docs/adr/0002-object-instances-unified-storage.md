# Unified `object_instances` Table with First-Class Child Rows

Every Object Instance — parent or child — lives as one row in a single `object_instances(tenant_id, object_type, external_id, properties JSONB, relationships JSONB, …)` table. Child objects (Payments under an Order, Reviews under an Order) are **not** nested inside the parent's JSONB; they are independent rows that reference the parent through their own `relationships` column. Queries that aggregate across children (e.g. `sum(payments.amount)`) run correlated subqueries / LATERAL joins against the same table.

## Why

Child objects must be first-class queryable on their own (e.g. "list negative Reviews", "export Payments for reconciliation"), so nesting them in the parent's JSONB would force rebuilding filtering and indexing for every child type. A fully normalized "shadow table per Object Type" variant was rejected for MVP because it requires the Mapping engine to double-write, and every tenant-created Object Type would trigger DDL — unacceptable for a configuration-driven platform at this stage.

## Consequences

- Indexing is selective: tenant admins flag properties as "filterable" or "sortable" when defining them; the platform creates expression indexes only for flagged properties on `(tenant_id, object_type, (properties->>'field'))`. Unflagged properties fall back to sequential scan with a UI warning.
- Decimal casts (`(properties->>'amount')::decimal(p,s)`) are generated by the query compiler from Ontology precision/scale declarations.
- Null propagation is handled at the compiler layer (see ADR 0001) — JSONB `null` and missing keys both coalesce to the type's zero value in aggregates.
- Write-heavy parents (e.g. an Order being updated rapidly) don't contend with child writes (Payment inserts), because children are separate rows.
- At 10⁶ rows per tenant (PRD §10.1 upper bound) aggregate queries may exceed the 10s target. The query compiler must estimate cost and offer async export or reject for queries above a threshold — tracked as a follow-up, not blocking MVP.
- Migration to shadow tables later (if performance demands) is feasible because the Ontology already declares the schema; the Mapping engine can start double-writing per-Object-Type tables without changing the public Query API.
