# Predicate / OntologyView / ObjectInstanceScope as Value Types in `@omaha/dsl`

The query subsystem is structured around three value types in `@omaha/dsl` that flow between the permission, ontology, and query modules: **Predicate** (a parsed + analyzed + bound DSL expression carrying its own scope), **OntologyView** (the runtime projection of an Object Type for a given tenant), and **ObjectInstanceScope** (the shape of the `FROM object_instances WHERE …` prefix). Two pure functions — `emit(predicate)` and `emitScope(scope)` — produce SQL fragments. `OntologyViewLoader` (request-scoped, lives in `OntologyModule`) is the single place that assembles `OntologyView` from Prisma. `PermissionResolver` (replaces the old `PermissionService`) consumes `OntologyView` to produce already-bound `Predicate[]`. `QueryPlannerService` consumes `Predicate[]` and stitches them into one SQL statement; it never substitutes templates as strings and never hand-codes `tenant_id`/`deleted_at`.

## Why

The previous architecture had three independent symptoms of the same shallow split: (a) row-level permission strings were re-parsed inside the planner via regex `:userId` substitution + manual `'...'.replace(/'/g, "''")` quoting — the only line standing between an actor's data and SQL injection; (b) the soft-delete invariant was hand-typed in four separate raw-SQL paths (planner WHERE, planner permission compile, QueryService include fetch, DSL compiler subqueries) so the Prisma monkey-patch held no actual hot read path; (c) `OntologyType` and `ObjectRelationship` rows were re-queried on every `plan()` call. Splitting into Predicate / OntologyView / Scope concentrates each concern: Predicate carries everything a SQL fragment needs; OntologyViewLoader is the only place that touches Prisma for shape; emitScope is the only place that knows how `object_instances` is queried. Locality goes up at every site.

## Consequences

- **No string-level template substitution exists in the codebase.** `:userId`/`:userRoleId`/`:userTenantId`/`:now` are bound as `param` AST nodes inside `PermissionResolver`; they reach the database as `$N` parameters. The whitelist is enforced at one place.
- **`emitScope` is the single authority for `FROM object_instances WHERE …`** — adding time-slice queries (V1.1: `deleted_at <= cutoff`) is a one-place change. Forgetting the filter would require deliberately not calling it; raw SQL bypassing it is now an architectural rule, not a hidden hazard.
- **`PermissionService` is gone.** `PermissionResolver.resolveOrThrow(actor, resource, action, objectType) → { allowed, allowedFields, predicates }` is the single resolver entry point. Future Action preconditions and NL-agent permission checks reuse it.
- **OntologyView is request-scoped and cached.** `OntologyViewLoader` (NestJS `Scope.REQUEST`) hits Prisma at most once per (tenant, objectType) per request. Resolver and Planner share the same view inside one request.
- **The `effective_permission_filter` audit field stores serialized Predicate JSON** (`{ ast, params }`), not a SQL-leaning rendered string. Replay/explain tools can deserialize it.
- **Predicate is serializable.** NL → Plan generation (ADR 0005), Action preconditions (ADR 0004), and offline Plan replay all consume the same shape. This is why Predicate is a value type, not a class with methods.
- **Predicate types live in `@omaha/dsl`; OntologyViewLoader and PermissionResolver live in core-api.** Pure types in the inner ring; I/O adapters in the outer ring. Future agent-worker can import `@omaha/dsl` to construct/inspect Predicates without pulling in Nest/Prisma.
- **Test surface improvement**: each of the new value types is independently unit-testable via its emit function; `OntologyViewLoader` and `PermissionResolver` are testable through their public methods on real Prisma in e2e. The cross-cutting "did I forget `deleted_at IS NULL`?" question disappears as a possible bug class, replaced by "did I call emitScope?" which is a code-review check.
- **Migration path forward**: any new read of `object_instances` (e.g. Action handler dry-runs in V1.1, agent-worker batch ops) must construct a Scope and route through `emitScope`. The `INCLUDE_DELETED_SYMBOL` ORM escape hatch remains for audit replay only.
