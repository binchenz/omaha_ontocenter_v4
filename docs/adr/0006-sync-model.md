# Sync Model: Full-Rebuild Default, Incremental Opt-In, Soft-Delete

> **Status: Draft — not yet implemented.** Schema tables (`SyncJob`, `Connector`, `ObjectMapping`) and stub modules exist but no sync execution logic has been written. This ADR records the design intent for when the sync engine is built.

MVP supports two sync strategies per `ObjectMapping`: `full` (default) and `incremental` (opt-in). A `full` sync treats each run as a world snapshot — upserts every row by `external_id`, and soft-deletes Object Instances that were present on the previous run but absent this time. An `incremental` sync requires the tenant admin to declare a watermark column on the source table (typically `updated_at`); it pulls only rows newer than the last stored watermark, upserts them, and performs **no** delete detection — removing rows in `incremental` mode requires a manual "Full Resync." Soft deletes are mandatory: every `object_instances` row gets a nullable `deletedAt`, and queries are required to filter `deletedAt IS NULL` at the compiler level. CDC / binlog subscription is explicitly deferred to V2.

## Why

PRD §5.2's data sources span CSV, Excel, MySQL, and PostgreSQL. File-based sources are naturally full-rebuild, so making `full` the default lets half the connector types work out of the box without admins designing a watermark strategy on day one. `incremental` is kept available because large SMB order tables make full scans painful at the `*/30 * * * *` cadence PRD §7.1 describes, but it's opt-in because its delete-blindness is a real data-quality hazard — admins must accept that cost explicitly by flipping the switch and wiring a cron-based "Full Resync" for compensating cleanup. Soft deletes unlock the time-slice query semantics that are the PRD's core differentiator (§9: "yesterday before 19:00"); a physical delete would make historical replay impossible because the row referenced by the time window would be gone.

## Consequences

- **Schema additions**: `ObjectMapping.syncStrategy` (`'full' | 'incremental'`), `ObjectMapping.incrementalCursorField` (nullable, required if strategy is `incremental`). `SyncJob.strategy`, `SyncJob.cursorBefore`, `SyncJob.cursorAfter`, `SyncJob.rowsDeleted`. `ObjectInstance.deletedAt`. `ObjectInstance.sourceRef` upgraded to `{ connectorId, tableName, sourceRowId, lastSyncJobId, lastSyncedAt }`.
- **Query-layer obligation**: every read of `object_instances` must filter `deletedAt IS NULL` by default. The enforcement gate is the **`ObjectInstanceScope`** module (in `@omaha/dsl`) and its `emitScope()` function: all read paths — the query planner's main WHERE, the planner's permission Predicates, the derived-property DSL compiler's `exists`/`count`/aggregate subqueries, and QueryService's `include` batched fetch — route through it. Bypass requires an explicit `includeDeleted: true` flag on the scope value, used only by audit replay. The Prisma `objectInstance.{findMany,findFirst,count,aggregate}` runtime patch in `PrismaService` remains as belt-and-braces for any ORM-path consumer, but it is not the primary defence — `emitScope` is. Forgetting the filter is an information leak, so it is a system-level invariant, not a per-service concern.
- **Transaction strategy for `full`**: MVP uses a single-transaction upsert + soft-delete per SyncJob. This caps MVP at tens of thousands of rows per run. Above that, the sync enters batched mode with savepoints — but only when the limit is hit; no premature optimization.
- **Failure semantics**: a SyncJob either fully commits or writes nothing. Partial writes never land in `object_instances`. Retry is manual (admin re-runs). Auto-retry/backoff is `agent-worker` work deferred to V1.1.
- **`incremental` admin contract**: the manager must explicitly declare which source column is the watermark. The platform does not infer it. If the column is absent, the strategy can't be selected. Delete detection is documented as the admin's responsibility — the UI surfaces a "Full Resync" button and a "last full resync" timestamp prominently.
- **Downstream for Plan 4**: the Derived Property compiler can assume Object Instances are monotonically non-deleted within a query scope and that soft-deleted rows are filtered upstream. Time-slice semantics ("was this row deleted before cutoffTime?") is a V1.1 query feature — MVP queries always see current-live data.
- **CDC deferral**: Debezium/Maxwell deployment is its own platform-level project (additional service, connector management, schema drift tolerance). Explicitly out of MVP and V1.x; a V2 ADR will supersede the "incremental only" half of this decision when that arrives.
