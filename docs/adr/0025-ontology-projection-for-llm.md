# OntologyProjection: the Ontology, not its Schema, is what the LLM sees

> **Status: EXPERIMENT RECORD — design partially REVERTED.** The full `OntologyProjection` redesign described below was implemented and then reverted: live-LLM behavioral testing showed it regressed the drama-query baseline 9/9→7/9. The de-specification (abstract tool descriptions, placeholder field names) was rolled back; concrete few-shot field anchors were kept. The structural-affordance follow-up proposed here as "the real fix" became Axis A and was itself **refuted** in the product pipeline (see [[0026-enum-constrained-tool-parameters-axis-a]]). The capability gap this ADR circled (cross-object aggregation) was later closed by a different mechanism (see [[0027-cross-relationship-aggregation-spike]]). This document is retained for the reasoning trail, not as a description of current architecture.

The Agent's system prompt presents the tenant's Ontology to the LLM through a single seam, **`OntologyProjection`** (a pure presenter in core-api). It produces two views of the *same* resolved Ontology: a small, always-resident **map** (`toResidentMap`) and an on-demand **detail** view per Object Type (`toTypeDetail`, served by the `get_ontology_schema` tool). The map carries the *shape* of the world — Object Types with a one-line description, Links with cardinality, the *names* of Derived Properties (the function layer) and Actions (the kinetic layer) hanging off each type, and each visible property as `name:type ✓↕[unit]` **without** its description. The detail view carries the *semantics* — each property's full, untruncated `description` (the ADR-0023 scale/navigation/sibling-boundary annotations), Derived Property signatures with descriptions, and Action names. Which properties are visible at all is decided by a consumer-neutral fact on the property — `internal: boolean` — not by the query engine's `filterable`/`sortable` capability flags.

## Why

Before this decision, the LLM never saw the Ontology — it saw the Ontology flattened into a SQL table listing. Three coupled symptoms of one architectural fault:

1. **The query engine shaped the Ontology.** `getSchemaSummary` admitted a property to the LLM's world only via `.filter(p => p.filterable || p.sortable)` — i.e. only if the SQL engine could put it in a `WHERE`/`ORDER BY`. On the drama-co reference ontology this admitted 38 properties and silently excluded 14. The excluded set mixed genuine ETL scaffolding (`from_string`, `character_name_raw` — "raw name before ID resolution") with the ontology's *highest-value* content (`tags` = genre, `revelations` = plot twists, `key_events`, `peak_chapter`). The gate's criterion — "can this go in SQL" — is uncorrelated with "does this carry business meaning," so it误伤 both directions. The output was literally headed `数据模型:` ("data model") — the term CONTEXT.md's Ontology entry lists under `_Avoid_: Schema, Model`.

2. **The annotation investment was destroyed in transit.** `MAX_DESC = 50` truncated every field `description` to 50 chars. On drama-co, 39 of 60 descriptions (65%) exceeded that. The truncation kept the sentence head and cut the tail — but the ADR-0023 disambiguation payload (the scale comparison, the "找 X 用此字段" navigation clause, the sibling boundary) lives in the tail. The single mechanism that exists to disambiguate `avg_tension` from `emotionalTone` was being deterministically deleted. CONTEXT.md defines Semantic Annotation as "consumed by the LLM via schema summary injection **to disambiguate**" — the code was defeating its own stated purpose.

3. **The function and kinetic layers were absent.** `getSchema` fetched `derivedProperties` but `getSchemaSummary` never rendered them — and even the fetch dropped everything but `name/type/label`, discarding the `description`/`expression`/`params` that `DerivedPropertyDefinition` already carries. Actions (`action_definitions`, fully modeled per ADR-0004) were not queried at all. In Palantir terms the platform shipped only a partial *semantic layer* and none of the *kinetic layer* (Functions + Actions) — so the Agent could only read, and the otherwise-correct Tool/Skill骨架 (ADR-0008) had nothing but `query_objects` to orbit, which is why query sat at the center and the Ontology became its configuration.

The deeper fault under all three: **the Ontology's representation to the LLM was shaped by its most demanding consumer (the SQL query path) instead of by the domain.** The fix inverts the coupling — the Ontology is the stable center; each consumer projects from it.

We split along **structure vs. semantics**, not along **everything vs. retrieval**, because the two naive options both fail. Rendering full descriptions resident ("thick map") hits the `PROMPT_BUDGET_WARN` (4000 tokens) on an 8-type ontology — descriptions are 65% of the bytes. A names-only "thin map" forcing a `get_ontology_schema` round-trip before *every* query doubles latency on the ~80% of queries that never needed disambiguation. The骨架-map keeps common queries zero-round-trip (the LLM plans from `status:string✓`, `amount:number✓↕[元]`) and pays the round-trip only when it must disambiguate by *meaning* — which is exactly when the detail is needed. `MAX_DESC` then disappears rather than moving: the resident layer never holds a description to truncate; the full text exists only in the detail layer, where nothing is cut.

`internal: boolean` records a **fact about the property** ("this is a structural/technical field, not a business attribute"), not a **policy about a consumer**. An earlier draft named it `aiHidden` — rejected: that names one consumer, so the next consumer (a UI detail page that also wants to hide `from_string`, an export) forces `uiHidden`, `exportHidden`, … a boolean per consumer. `internal` is shared by all consumers, each deciding what to do with it; the projection's *policy* ("drop internal fields") lives in `OntologyProjection`. The admission default is inverted to **visible-unless-internal** (blacklist), because the失败 modes are asymmetric: a forgotten `internal` flag leaks `from_string` as a few tokens of noise (visible, fixable); a forgotten `filterable` under the old gate silently erased `revelations` from the Ontology (a loss of expressive power). Default-complete-minus-noise matches Palantir's posture: the Ontology is complete by default; the projection subtracts noise rather than adding back fields.

## Consequences

- **`OntologyProjection` is the single seam between the resolved Ontology and the LLM.** It is a pure presenter in core-api (no NestJS, no Prisma): in = a resolved Ontology (the `getSchema` product), out = strings. `getSchemaSummary` collapses to a one-line call into it; the symbol legend (`✓=可过滤 ↕=可排序 […]=单位 {…}=字段描述`) moves out of QuerySkill's 注意事项 and is emitted *next to the data it decodes*, ending the legend-split that [[prompt-skill-assembly]] flagged. Three consumers now share one view: the orchestrator's resident prompt, the `get_ontology_schema` tool, and (later) Action rendering. It passes the deletion test — remove it and this logic regrows across all three callers.

- **`internal: boolean` is added to `PropertyDefinition`** (in `@omaha/shared-types`), default `false`. ETL scaffolding fields (`*_string`, `*_raw`) are marked `internal: true`; the existing three seeded ontologies must be back-marked. It is set on the same path as `description`/`unit`: the DataIngestionSkill infers it when generating raw fields ([[project_data_ingestion]]); hand-authored ontologies set it explicitly. It is **one axis** — "non-business" — and is deliberately *not* overloaded with "sensitive" (the ADR-0024 schema-exposure concern), which is an orthogonal axis (business but not externally safe) and would get its own flag if needed.

- **`filterable`/`sortable` stop gating visibility.** They survive only as capability markers (`✓`/`↕`) on already-visible properties, telling the LLM what can go in `WHERE`/`ORDER BY`. Visibility is now governed by `internal` alone.

- **Read-only scope is honored precisely.** This decision delivers *representation fidelity only*: descriptions untruncated in the detail layer, Derived Properties visible (names in the map, signatures+descriptions in detail — `getSchema`'s projection is widened to stop dropping them), Links carry cardinality. Actions appear in the map as **names only** ("能力槽") so the LLM knows a type is actionable, but are **not callable and carry no detail** — they have no `description` field today (only `label`) and the Agent has zero Action execution path. Making Actions callable is the kinetic-layer step (ADR-0004 lifecycle), explicitly deferred. The 1+2 vision is preserved by *reserving the slot*, not filling it.

- **The detail layer reuses the existing `get_ontology_schema` tool**, which becomes parameterized (`objectType?`) — no-arg returns the map-equivalent, with-arg returns `toTypeDetail`. The retrieval leg Palantir-style ([[prompt-skill-assembly]] deferred) is thereby half-built already; this ADR completes its contract rather than inventing a new tool.

- **The budget pressure that triggered this is resolved structurally, not by tuning a constant.** Moving the 65%-of-bytes descriptions out of the resident layer cuts resident size to roughly a third; the `maxTypes` overflow already routes to `get_ontology_schema`. `PROMPT_BUDGET_WARN/ERROR` remain as the guard.

- **Not in scope, recorded for later:** (a) `from_string`-class fields arguably should not be `ObjectType` properties at all — they are ingest staging products leaking into the Ontology. Marking them `internal` is the cheap fix; removing them from the Ontology is a separate ingest-pipeline cleanup. (b) `tags`/json-array fields are now visible but still raise `PROPERTY_NOT_FILTERABLE` on direct filter; QuerySkill already instructs "改用 search," so visible-but-not-directly-filterable is a pre-handled known state. (c) Tenant-specific tool-face generation (one tool per Object Type/Action) is the maximal Palantir form and remains deferred — it would explode tool count into the same budget from the tools side.

## Extension: capability honesty and a single source of truth

The first cut made the Ontology *legible* to the LLM (semantics no longer truncated, derived/links/internal projected). A follow-up review against the question "is the Ontology *fully* perceivable?" found three holes that all trace to the same fault the original ADR set out to fix — a **second source of truth about what fields/operations exist**, drifting from the Ontology:

1. **Tenant field names in tool descriptions — and why we kept them.** `aggregate_objects` teaches the LLM by example using concrete drama-tenant fields (`character_name_raw`, `pace_type`, `total_chars`). The instinct (matching [[prompt-skill-assembly]]'s `delivery_*` cleanup) was to scrub these to placeholders for a single source of truth. **We tried it and reverted it, because live-LLM behavior testing refuted it.** On `drama-query.e2e` scenario S4 ("哪部剧特写镜头最多" — a group-and-count-max query that must route to `aggregate_objects`), the concrete-example version on `main` passed ~2/3 (retry-backed); an abstract "decision rule" rewrite scored 0/3; placeholder examples scored 1/3. The concrete field names are not merely tenant-coupling noise — they are **few-shot anchors** carrying real disambiguation hit-rate. Scrubbing them traded the LLM's actual accuracy for architectural purity, and the data said no. What we *did* keep: the one example using `character_name_raw` was deleted (it both contradicted that field's new `internal: true` mark and demonstrated a query that should go through the `BookCharacter` object), and a field-name-free "数量最多的前 N 个 → groupBy + count + orderBy desc + maxGroups" shape was added so the count-ranking pattern survives without binding a tenant field.

   **The real lesson, recorded for the follow-up:** prose (tool descriptions, examples, schema summaries) is a *probabilistic* way to convey affordance — the LLM pattern-matches and guesses, and no wording is reliably correct (S4's three-way result is the proof). The principled fix is not better prose but removing prose from the loop: an Ontology-generated **tool face** where field parameters are `enum`-constrained to the type's real fields and capability (filterable/sortable/numeric) is enforced *structurally* rather than described. That is a separate, larger architecture line (its own ADR + PRD), deliberately not crammed into this projection change. This ADR's gap-A outcome is therefore: **revert the de-specification, stop the S4 regression, and escalate the structural fix as the next line.**


2. **Derived properties visible but their usability silent.** The map lists derived-property names, but nothing told the LLM *which operations are legal on them*. Verified engine reality: filter ✅ (via the `derivedProperty` filter field, a separate path from `field`), sort ❌ (not in `sortableFields` → `resolveSort` **silently falls back to `created_at`**), aggregate metric ❌ (not in `numericFields` → `METRIC_INVALID_FIELD_TYPE`). Silent sort fallback is the worst state — not "can't sort" but "the LLM believes it sorted and got wrong results quietly." Fix: the `query_objects`/`aggregate_objects` filter schemas gained `derivedProperty` + `params` (the canonical `QueryFilter` already supported them; only the tool JSON-schema omitted them), and `toTypeDetail` now **declares each derived property's capability honestly** — filterable, not sortable, not a metric — driven by real engine behavior, not invented.

3. **A dead overflow pointer.** When types exceeded `maxTypes`, the map said "更多请调用 get_ontology_schema" — but no-arg `get_ontology_schema` returned the *same truncated map*, so the 16th type was unreachable in a loop. Fix: the overflow line now **names** the overflow types and instructs `get_ontology_schema('<name>')` to drill any of them; `maxTypes` raised 15→30 (a budget dividend from moving descriptions out — each map line is now shorter). The cap itself is kept: the resident layer is paid every turn, so unbounded residency would re-create the "thick map blows the budget" failure the original ADR rejected; the fix is overflow *reachability*, not removing the cap.

### Why this is the Palantir form, and what we deliberately did NOT do

The unifying principle (from Palantir Foundry/AIP's ontology design): **capability is declared in the Ontology and projected out; tools and prompts hold no field knowledge of their own.** A reviewer's first instinct — "just write the sort/aggregate limits into the tool descriptions" — was rejected for being the very disease (a second place that knows fields). Capability boundaries are now spoken by the projection (`toTypeDetail`), read by tools, never hand-written per tool.

We chose **capability honesty over capability expansion**. The alternative — actually making derived properties sortable/aggregatable by extending `resolveSort`/the metric path — is a *new query-engine feature*, not "making the Ontology perceivable," and would pull in ADR-0017's aggregation semantics. Deferred. Also deferred: unifying `field`/`derivedProperty` into one "just name the property, the Ontology resolves stored-vs-computed" entry (the fuller Palantir form) — it would change the `QueryFilter` contract and `ScopedWhere` parsing, a larger blast radius than exposing the existing `derivedProperty` entry.

### Consequences of the extension

- **Adding a tool or skill carries an architectural rule:** it must not embed tenant field names or hand-write per-field capability. Operation legality is sourced from the projection. This is now a code-review check, like "did you call `emitScope`?" (ADR-0007).
- **The LLM's belief about an operation equals the engine's real capability.** Silent-fallback-as-lie (derived sort → `created_at`) is replaced by an explicit "not sortable" declaration. The `sortFallbackReason` the planner already returns remains the runtime backstop.
- **Test surface:** `toTypeDetail`'s capability declaration and the overflow-naming are unit-tested in `ontology-projection.spec.ts`; the `derivedProperty` filter path was already covered end-to-end by `derived-property.e2e-spec.ts`, so exposing it via tool schema added no new engine path to test.
